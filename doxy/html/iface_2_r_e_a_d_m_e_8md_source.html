1 Wire Cell Interfaces
2 ====================
3 
4 Overview
5 --------
6 
7 This package provides a set of classes which define the interface to
8 Wire Cell. Software from the other Wire Cell packages that provides
9 Wire Cell functionality do so through implementing these interfaces
10 and by calling into them. Outside "client" code that wishes to use
11 Wire Cell code directly should do so by compiling and linking against
12 this package. In general there should not be direct compilation or
13 linking against any higher level package in Wire Cell.
14 
15 Component Interfaces
16 --------------------
17 
18 Major facets of Wire Cell inherit from a subclass of
19 `WireCell::Interface` (which is actually provided by `util`). Such
20 classes have a (non-const) `shared_ptr` defined by the base class as
21 these facets tend to be shared by multiple client code. The concrete
22 implementation of an Interface register through the
23 `WireCell::NamedFactory` mechanism which allows dynamic construction.
24 
25 Data Interfaces
26 ---------------
27 
28 The data objects which Wire Cell produces and operates on are defined
29 through interface classes as well. Their interfaces do not inherent
30 from `WireCell::Interface` as components. Rather, all data interface
31 classes inherit from `WireCell::IData<Data>`, which is templated on
32 the data interface class itself. See for example `WireCell::IWire`.
33 
34 Every data interface class has a number of types defined including a
35 (const) `shared_ptr`, a base iterator as well as an abstract iterator
36 and iterator range. Also defined are types for Wire Cell signal/slot
37 mechanism which is described more below.
38 
39 Some specific data interface also define some types for convenience
40 such as providing a common vector collection. These types follow a
41 common naming convention, for example: `WireCell::IWireVector`,
42 `WireCell::ICellVector` and `WireCell::IPlaneSliceVector`.
43 
44 There are also interfaces for sequences of data to provide
45 light-weight memory views which act like simple iterators but which
46 may hide more complex data structures. For example, cells can be
47 stored in a memory-efficient graph structure which also provides fast
48 associations. For simple iteration it is best to reuse that structure
49 while providing "familiar" iterators.
50 
51 Some of the important data interfaces include:
52 
53 * **wire** a segment of wire with physical endpoints and associations to channel and an index into its plane of wires.
54 
55 * **cell** a region of space in the wire plane associated with one wire from each.
56 
57 * **blob** a collection of cells which itself implements the cell data interface.
58 
59 * **wire summary tiling** implement common queries on wires and cells.
60 
61 * **wire plane and channel slices** implement access to information about charge in a slice in time across either a plane of wires or a detectors readout channels
62

64 Signal/Slot
65 ===========
66 
67 Wire Cell implements a pattern called "data flow programming" (DFP).
68 Instead of just calling functions with arguments and accepting their
69 returned results, DFP allows for high level "flow networks" to be
70 defined by connecting together functional *units* (objects) by
71 attaching one unit's output to another unit's input.
72 
73 The reasons to follow the DFP paradigm are:
74 
75 * minimize the amount of data that must be buffered into memory at any one time.
76 
77 * reduce the coupling between units.
78 
79 * support multiple computing architectures (single-processor,
80  multi-processor, massive parallelism, CPU/GPU) with maximal code
81  reuse.
82 
83 
84 Boost's Signal/Slot `boost::signals2` mechanism is used. Components
85 which accept DFP input stream have a signal object which accepts
86 connections of slots matching a particular functional prototype.
87 Depending on the nature unit it may be that only a single connection
88 is allowed or that multiple ones can be made and possibly in a
89 predetermined order.
90 
91 When the unit requires new data it "fires" (calls) its signal which
92 calls its slots and returns to the calling unit their return value(s).
93 
94 Typically a slot is implemented as a `IData::pointer operator()()`
95 method which lacks any arguments. But, in some cases such as
96 `WireCell::Fanout` slot arguments may be required and are passed in
97 through the signal call. See also `WireCell::Addresser` which binds a
98 argument to pass at construction time which may be used to access a
99 particular "jack" of a `WireCell::Fanout`.
